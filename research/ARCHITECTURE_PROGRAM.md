# Architecture Program

## A. Program Hypothesis
RepoTrace is a research instrument for testing a strategic asymmetry: under the current LLM-agent equilibrium, durable search advantage comes from better promise selection and sustained semantic focus, not from marginally better shallow local search.

## B. Strategic Context
Shallow local search is increasingly commoditized. The scarce step is hidden-spec reconstruction: selecting the right promise boundary, holding it stable, and falsifying it with disciplined pressure. Repo size is usually secondary to objective ambiguity and semantic state-space complexity.

The program is informed by process-level architecture ideas: working memory, predictive routing/gating, replay/consolidation, and typed memory plus control. These are design inspirations for search leverage, not commitments to immediate runtime automation.

## C. Authority Model
- Architecture direction is set by explicit architectural reasoning and review, not feature-branch convenience.
- Codex may autonomously update implementation details and governance artifacts within the current architecture/control plane.
- Changes to high-level thesis, ontology, invariants, authority boundaries, or stage map require explicit architectural review and/or decision-log updates.
- Division of labor:
  Human steers hypothesis, authority boundaries, and acceptance criteria; Codex executes implementation and reconciliation with explicit traceability.

## D. Lifecycle / Stage Map
- Stage 0: context and promise derivation.
- Stage 1: promise selection and adjudication.
- Stage 2: promise decomposition (`promise x interaction family x consistency boundary`).
- Stage 3: promise-scoped probing.
- Stage 4: outcome and closure.
- Stage 5: cross-promise and higher-order expansion.

## E. Success Conditions
Real progress means measurable search leverage where shallow agents fail:
- better promise stability before probing,
- faster and cleaner falsification/closure decisions,
- lower objective drift,
- higher decision quality under ambiguous objectives,
- favorable signal even after discounting retrieval-shaped proxy gains.

Architecture elegance alone does not count as success without search/decision improvements.

## F. Kill Criteria / Strong Falsifiers
Program direction is likely wrong/low-value if:
- promise selection remains unstable despite sustained effort,
- operators consistently bypass promise-scoped flow for better outcomes,
- measured gains come mainly from retrieval-style proxies,
- cross-promise failure modes dominate and remain systematically missed,
- artifact/process burden grows faster than search leverage.

If these hold across multiple incident families and objective profiles, the current program boundary should be reconsidered.

## G. Provisional Mechanisms
Core commitments:
- promise-first thesis,
- explicit semantic decomposition,
- typed memory/control boundaries,
- evidence/witness grounding.

Provisional instruments (replaceable if they underperform):
- touch maps,
- check library shape,
- work packets,
- parts of promise-manual metrics,
- portions of derivation/review path.

These mechanism shapes are instrumental, not sacred; the non-negotiable commitment is promise-first semantic search quality and closure quality.

## H. Later-Stage Expansion Ladder
- upstream strengthening (promise derivation/selection discipline),
- higher-order semantics and cross-promise modeling,
- probe execution refinement,
- controlled autonomy under explicit governance,
- late formalization (including invariants) after stable value signals.

## I. Update Rule
Update this file only when program-level assumptions change: hypothesis, strategic context, authority boundaries, stage map, success/kill criteria, or provisional-vs-core boundaries. Do not update for routine branch mechanics or local implementation detail.
